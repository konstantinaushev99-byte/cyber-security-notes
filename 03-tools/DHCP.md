# DHCP

# Что такое DHCP?
DHCP - это протокол для автоматической настройки адресов для всех устройств в нашей сети.

---

Процесс DORA (4 шага)

| Шаг | Название | Что происходит | Тип трафика |
|:---:|:---------|:---------------|:------------|
| D   | Discover | обноружение DHCP серверов | Broadcast |
| O   | Offer    | Сервер предлогает ip-адреса | Broadcast или Unicast |
| R   | Request  | Запрос предложеного адреса  | Broadcast |
| A   | Acknowledge | Сервер подтверждает и выдаёт параметры | Broadcast или Unicast |

Для запоминания: DORA — как открытие двери: Discover (поиск), Offer (предложение), Request (просьба), Acknowledge (подтверждение).

---

# DHCP-relay(DHCP-помощник)

Проблема: DHCP-сервер в другой подсети, а Discover — широковещательный (не проходит через роутер).

Решение: Настроить на роутере DHCP Relay (или IP Helper). Роутер пересылает DHCP-запросы в подсеть с сервером.

Как работает:
1. Клиент отправляет Discover (Broadcast)
2. Роутер перехватывает, пересылает серверу как Unicast
3. Ответ от сервера роутер отправляет клиенту

# Атаки на DHCP (важно для SOC)
 Rogue DHCP Server

то это: Злоумышленник поднимает свой DHCP-сервер в сети.

Как работает атака:
1. Клиент отправляет Discover
2. Первый ответивший сервер выигрывает (обычно тот, кто быстрее)
3. Если rogue-сервер ответил первым — клиент получит:
   - Подставной шлюз (весь трафик идёт через атакующего)
   - Подставной DNS (перенаправление на фишинговые сайты)

Защита:
- DHCP Snooping — настройка на коммутаторе, которая блокирует недоверенные DHCP-серверы
- Отключать неиспользуемые порты

---

# DHCP Starvation

Что это: Атакующий создаёт тысячи запросов с разными MAC-адресами, чтобы истощить пул IP-адресов.

Результат: Новые устройства не могут получить IP.

Защита* ограничение количества MAC-адресов на порту (port security).

---

# Для собеседования (одной строкой)

DHCP выдает IP через 4 шага: DORA (Discover, Offer, Request, Acknowledge). Rogue DHCP — атака подставным сервером. DHCP Snooping — защита от неё. DHCP Relay позволяет клиенту и серверу быть в разных подсетях.

# В терминале Debian (моя виртуальная машина)

# 1. Посмотрел IP-адрес (убедился, что он получен динамически)
ip a
# Вывод: inet 192.168.3.114/24 ... scope global dynamic

ip route
# Вывод: default via 192.168.3.1 ... proto dhcp
# proto dhcp = маршрут получен через DHCP

# 3. Посмотрел DNS-сервер
cat /etc/resolv.conf
# Вывод: nameserver 192.168.3.1
# DNS-сервер тоже выдан DHCP (или роутером)
